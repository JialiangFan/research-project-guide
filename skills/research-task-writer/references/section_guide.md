# 分节写作指引：每一节怎么写到位

下笔拿不准某节怎么写时读这里。每节给"职责 / 好的样子 / 常见反例"。范例多提炼自一个成熟的重型 task（MuJoCo 装箱场景设计）。

## 总原则

- **每一节都要有信息量。** 只剩套话的节就精简、合并或删掉。宁可短而实，不要长而空。
- **能表格就表格。** 交付物、代码位置、状态这类多字段信息，表格比长段落好读、好维护。
- **写给三个月后的人。** 他不记得上下文，也没读过代码。术语第一次出现给一句解释，路径写全。
- **真实优先。** 路径、命令、指标只写真实存在或确定的；不确定就 `TBD` 并说明缺什么。

## 状态

**职责**：让人 3 秒看出这个任务现在能不能用、新不新。

**好的样子**：

```
状态：已完成
最近更新：2026-06-23
一句话进展：carton 装箱场景已落地（几何+资产+导出+clean/fault eval），正式 eval matrix 待登记到 runs/。
```

一句话进展点出"做完了什么 + 还差什么"，比单一个"已完成"信息量大得多。

**反例**：只写 `状态：进行中`，没有日期、没有进展句——读者无法判断是昨天还是三个月前停在哪。

---

## 目标

**职责**：用一两句说清"做什么、达成什么、边界在哪"。

**好的样子**：先说重设/要做的事，再说约束。例：
> 把场景从"桌面散放杂物 → 2×2 托盘"重设为"从运输纸箱逐个取同款方盒 → 放进零售展示盒"的上架任务；场景仍须能 step physics、渲染 RGB-D，并保留原有 contract/scenario schema。

注意它同时给了**正向目标**和**不变约束**（仍要满足什么），边界很清楚。

**反例**："优化场景，让它更好。"——什么叫更好？没有可判断的边界。

---

## 背景

**职责**：解释为什么需要这个任务、它在项目里的位置（上游依赖、下游服务）。

**好的样子**：点明动机和依赖关系，必要时附参考图/直觉来源。轻量任务一两句即可。

**反例**：把整个项目的来龙去脉复述一遍。背景只服务"理解这个子任务为什么存在"，不是项目综述。

---

## 技术路线

**职责**：方法 + 输入 + 输出 + 关键取舍。这是技术读者最关心的一节。

**好的样子**：
- `方法`：不只说"用 MuJoCo"，还说**怎么用**和**为什么**——"场景仍由 runtime generator 生成 MJCF，不手写独立场景文件；在同一 schema 上替换容器、资产池、slot layout"。带上关键设计取舍。
- `输入`：列具体——schema 名、mesh 编号、缩放系数、seed、baseline 资产。
- `输出`：列具体产物形态——生成的 MJCF、命名场景、standalone xml + manifest + 预览图、trace。

**反例**：方法只写一个框架名（"用 MuJoCo / 用 PyTorch"），没有任何"怎么用、为什么这么选"。框架名不是技术路线。

## 实现计划

**职责**：代码实现在哪里、改/增什么。让读者能直接跳到对应文件。

**好的样子**：用表格列**真实文件路径 + 作用 + 状态**：

| 模块 / 文件 | 作用 | 状态 |
|---|---|---|
| `src/contractguard/pgtray/scenario.py` | carton 资产、场景 builder、难度梯度 | 已完成 |
| `src/contractguard/pgtray/mujoco_source.py` | 渲染 source bin / display fixtures | 已完成 |

也可以先用编号列表写"要做的改造步骤"，再用表格记落地状态——重型任务两者都有更清楚。

**反例**：只写"实现场景生成逻辑"，不给文件路径。读者无从下手。路径写错或编造比不写更糟——会把人引向不存在的文件。

---

## 指标

**职责**：列出关心的指标并**给出定义**，保证跨 run、跨人可比。

**好的样子**：每个指标一行定义：
- `success_rate`：成功 episode 数 / 总 episode 数
- `interpenetration_count`：对象与桌面/容器的初始穿插数量
- `fault_attribution_accuracy`：故障场景是否触发预期 recover/block reason

**反例**：只写指标名不写定义（"看 success_rate 和 accuracy"）。不同人对同一个名字的算法可能不同，结果就不可比。纯写作类任务没有量化指标，整节省略即可，别硬凑。

---

## 交付物

**职责**：交付物表（含真实路径）+ 能跑的产出命令 + 验证命令。"完成"长什么样在这里定死。

**好的样子**：表格四列 `交付物 | 路径 | 状态 | 说明`，逐项点明产物或代码在哪、什么状态。例如把"场景生成器""fixture 渲染""standalone 场景文件""可视化 QA 图""单测""eval 证据"分别列一行，状态分别标 `已完成 / Partial / 临时证据`——**诚实标注半成品**比一律写"已完成"可信得多。

命令分两类写清：
- 产出命令：生成交付物的命令（如 `python scripts/export_pgtray_scenes.py`）。
- 验证命令：查看/核对的命令（如 `cat runs/.../metrics.json`）。

**反例**：
- 交付物只写"代码和结果"，无路径无状态。
- 把每次 run 的精确命令行、git commit、env 抄进来——这些属于 `run_config.json`，抄进 task 文件必然过期失同步。

---

## 复现入口

**职责**：把读者导向真正的复现真相，而不是在 task 文件里重抄一遍。

**好的样子**：
> - 发起新 run：见 `../运行命令.md`
> - 完整命令 / 代码版本 / 环境 / 数据：见对应 `runs/.../run_config.json`
> - 结果登记：见 `../实验结果.md`

**反例**：在 task 文件里贴一大段"复现步骤"，包含写死的 git commit、conda 环境、数据 hash。这些一旦换 run 就错；它们的家在 `run_config.json`。

**写作类任务变体**：没有 runs/ 时，这节改成"如何核对产物"——例如"改动见 AAAI draft 的 §3.2；对照下面验收标准逐条检查"。

---

## 验收标准

**职责**：可勾选、可客观判断的完成条件。

**好的样子**：每条都能被明确判定真/假：
- [ ] source bin + display 在 MJCF compile 且无穿插
- [ ] clean episode `success_rate=1.0`、`external_execution_error_total=0`
- [ ] 每个命名场景导出 standalone `.xml` + manifest + 预览图

**反例**："场景做得不错""效果满意"——无法客观判定，永远扯皮。验收标准是闸门，必须可证伪。

---

## 阻塞与边界

**职责**：把"已有的 / 这次做的 / 还没做的"分开，划清当前边界。

**好的样子**：三段式——
> - 已有底座：MuJoCo 生成器、机器人/夹爪/物体加载、RGB-D、导出。
> - 本次落地：source bin / display fixtures、carton 资产、命名场景与故障、导出+预览。
> - 仍未做：多 seed eval matrix、force-level no-crush gate、真实贴图。

读者一眼知道能用什么、不能指望什么、下一步在哪。

**反例**：只写"还有些问题没解决"——哪些？阻塞具体是什么、卡在哪、需要什么才能解，要写明。

---

## 备注

**职责**：关联决策、设计取舍、偏离计划之处。维护任务记忆。

**好的样子**：
- 记关联决策："决策 18（运行时生成器=单一真相）继续成立"。
- 记偏离："本想用牙膏盒 mesh，实际改用 HOPE `obj_000014` 方盒，原因是需求要更接近正方形"。
- 过时内容用「（历史记录）」标注保留，不直接删——保住决策轨迹，别让后人重踩。

**反例**：把所有变更直接覆盖删除，文件只剩"最终态"。后人不知道为什么是这样、之前试过什么、为什么放弃。
