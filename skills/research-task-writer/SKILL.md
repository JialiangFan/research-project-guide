# 研究项目 task 文件写作指南

这个 skill 指导你为研究项目写出**清晰、可维护、可阅读**的 task 文件，统一放在 `docs/<project>/tasks/` 下。
目标读者是几个月后的自己和协作者：他们要靠这份文件搞清楚"这个子任务要做什么、用什么技术路线、代码在哪、产物是什么、怎么复现、怎么算完成"。

**输出一律全中文（含章节标题）。** 引用的英文 reviewer 原文、数学公式、代码标识符、路径、命令可以保留原文，但所有叙述性正文用中文。

---

## 四条核心原则

写之前先把这四条记在心里——模板只是骨架，这四条才是质量来源。

1. **全中文、可读。** 章节标题和正文都用中文。句子说人话，能用表格就别堆长段落。一个没读过代码的人，光看这份文件应该能复述出任务在做什么。

2. **docs 说"做什么/为什么/怎么验收"，runs 才是复现单一真相。** task 文件描述意图、技术路线、验收标准，并**指向** `runs/.../run_config.json`（那里才记录本次实际命令、git 版本、环境、数据）。task 文件**不抄录**这些易过期的细节——抄了就会和 runs/ 不一致。详见下面"与 docs/runs 的分工"。

3. **交付物要既能定位、又能复现。** 每个交付物写清**真实路径**（代码位置和产物位置），并配上**能直接粘贴运行**的产出命令和验证命令。读者据此能找到代码、跑出结果、核对产物。路径和命令必须真实存在或明确标注待定，绝不编造。

4. **可维护，随进展更新。** `状态` 和 `最近更新` 跟着实际进展改。计划变了、对象换了，就更新正文，并把过时内容用「（历史记录）」标注保留**而不是直接删**——保留决策轨迹，别让后人重踩坑。偏离计划要写进 `备注`（本想做 X、实际做了 Y、原因 Z）。

---

## 工作流程

1. **定位项目、沿用既有约定。** 先看目标项目的 `docs/<project>/` 目录：有没有 `README.md`、`运行命令.md`、`实验结果.md`、`tasks/` 下已有的 task 文件。**沿用**已有的命名（`NN_<task_name>.md`，如 `03_train_policy.md`）、编号和写法，保持一致。如果项目目录还不存在，按 `docs/<project>/tasks/` 结构新建，并提醒用户可能还要补 `README.md` / `运行命令.md` / `实验结果.md`。

2. **按标准模板填。** 用 `assets/task_template.md` 作骨架（章节清单见下）。**必填节**一定写；**可缩放节**按任务轻重决定保留还是精简（见"按任务轻重缩放"）。

3. **信息不全就显式标 TBD，不要编造。** 还没定的代码路径、命令、指标，写 `TBD` 或 `待定` 并说明缺什么，比填一个看似具体实则瞎编的路径要好得多——后者会误导读者去找不存在的文件。

4. **写完自检。** 对照文末"完成前自检清单"过一遍：四要素齐不齐、路径真不真、命令能不能跑、是不是全中文、有没有和 runs/ 重复抄录。

写每一节"好长什么样、常见反例"，需要时读 `references/section_guide.md`；想看完整范例（一个重型代码任务 + 一个轻量写作任务），读 `references/examples.md`。

## 标准模板：章节清单

完整可填空版本在 `assets/task_template.md`。各节一句话职责：

| 章节 | 必填? | 职责 |
|---|---|---|
| `# 任务 <编号>：<标题>` | 必填 | 一眼看出是第几个任务、做什么 |
| `## 状态` | 必填 | 规划中/进行中/阻塞/已完成/已废弃 + 最近更新日期 + 一句话进展 |
| `## 目标` | 必填 | 这个子任务要完成什么、边界在哪 |
| `## 背景` | 可缩放 | 为什么需要、依赖哪个上游、服务哪个下游 |
| `## 技术路线` | 必填 | **方法/输入/输出** + 关键设计取舍（用户四要素之一） |
| `## 实现计划` | 必填 | 代码实现在哪里、改/增什么——用表格列**真实代码路径**（四要素之一） |
| `## 指标` | 可缩放 | 实验类必填，给出指标**定义**保证跨 run 可比；纯写作类可省 |
| `## 交付物` | 必填 | 交付物表（含路径列）+ 产出命令 + 验证命令（四要素之二、之三）|
| `## 复现入口` | 必填 | 指向 `运行命令.md` / `runs/.../run_config.json` / `实验结果.md`（四要素之四）|
| `## 验收标准` | 必填 | 可勾选完成条件，**每条内联一条可执行「验证」命令**（治谎报完成）|
| `## Checkpoint 切片` | 必填 | 2–5 个有序纵切片，每片绑一条验收标准 + 报到点（治漂移）|
| `## 阻塞与边界` | 必填 | 已有底座 / 本次落地 / 仍未做 + `本任务不做（Non-goals）` + 越界处置（治理解偏差/过度工程）|
| `## 开放问题` | 必填 | 执行中冒出的越界 / 待决策问题往这堆，初始可空 |
| `## 备注` | 可缩放 | 关联决策、设计取舍、偏离计划之处 |

> **三道执行闸门**就嵌在上面三个加粗节里：`验收标准`（完成=证据）、`Checkpoint 切片`（中途报到防漂移）、`阻塞与边界` 的 Non-goals + 越界处置（范围栅栏）。它们让"完成不可被声称、意图不可被默认、范围不可被偷偷扩大"。派活时**配套附上**执行契约预设词（`~/.claude/harness/execution_contract.md`），把这三道闸门变成 agent 的实际行为。

## 用户四要素落在哪里（重点）

这个 skill 存在的根本理由，是保证每份 task 都能回答这四个问题。务必让它们清晰可见：

- **技术路线** → `## 技术路线`：`### 方法`（用什么技术/算法/模型/simulator/backend，关键取舍）、`### 输入`、`### 输出`。
- **交付物** → `## 交付物`：表格 `交付物 | 路径 | 状态 | 说明`，"完成"长什么样一目了然。
- **实现的代码位置** → `## 实现计划` 的代码表（`模块/文件 | 作用 | 状态`）+ `## 交付物` 的"路径"列。指向**真实存在**的文件/模块，让读者能直接跳过去。
- **可运行/复现的 shell 指令** → 两个层次：
  - `## 交付物` 里的**产出命令**（生成交付物的命令）和**验证命令**（查看/核对产物的命令），**能直接粘贴运行**。
  - `## 复现入口` 指向 `runs/.../run_config.json`——那里记录本次**实际**完整命令、git 版本、环境、数据，是复现的单一真相。

## 按任务轻重缩放

任务有轻有重。重型代码/实验任务（如搭仿真场景、训练评测）章节填满；轻量写作/分析任务（如改论文某节回应 reviewer）该精简就精简，别硬凑空话。

**轻量任务缩放规则**（仍**强制中文**，仍保留 `状态`/`目标`/`交付物`/`验收标准`/`阻塞与边界` 的 Non-goals）：

- `## 技术路线`：压成"思路 / 参考依据（如 reviewer 意见）/ 产出形态"，不必硬分方法/输入/输出三小节。
- `## 实现计划`：改成"要改哪些文件或论文章节"，列清楚位置即可。
- `## 复现入口`：没有 runs/ 的写作类任务，改成"如何核对产物"（如：在哪个 draft 里看改动、对照哪条验收标准）。
- `## 验收标准`：写作类没有 metrics.json，验证命令改成可跑的 `rg`/`diff` 检查（如 `rtk rg -q "<要求出现的论点>" paper/section.tex`），仍**每条绑一条能判真假的命令**。
- `## Checkpoint 切片`：轻量任务可压成 1–2 片或"一次到位"，但仍写出报到点（产出哪份 draft / 对照哪条验收标准）。
- `## 指标`：纯写作类可整节省略。

判断原则：**每一节都要有信息量。** 如果某节只剩套话，就精简或合并；如果某重型任务需要额外的领域小节（如 01 范例里的"场景设计""相机边界"），大方加——模板是地板不是天花板。

## 与 docs / runs 的分工（别抄重复）

仓库的复现架构是：**runs/ 是复现单一真相，docs/ 只指向不抄录。** 这直接决定 task 文件该写什么、不该写什么：

- **该写进 task 文件**：技术路线、代码位置、交付物路径、能粘贴跑的产出/验证命令、验收标准、设计取舍。
- **不该抄进 task 文件**（这些只属于 `runs/.../run_config.json`）：本次运行的**精确**完整命令行、`git_commit`/`git_dirty`/`git_branch`、`env`（python/依赖/GPU/快照）、`data`（路径/版本/hash）。task 文件里 `## 复现入口` 指过去就行。

为什么这样分：这些细节每次 run 都不同、最容易过期；抄进 task 文件必然和 runs/ 失同步。让 task 文件保持稳定的"规格 + 入口"，把易变细节留在 runs/。

> `## 交付物` 里的产出/验证命令是**代表性的、说明用途的**命令（"用这类命令能生成/核对它"），方便读者快速上手；`run_config.json` 里才是某次 run **实际执行**的精确命令。两者角色不同，不矛盾。

## 完成前自检清单

把这几条当作交付前的过滤器，不达标就回去补：

- [ ] **全中文**：标题和正文都是中文（除公式/标识符/路径/命令/引用原文）。
- [ ] **四要素齐**：技术路线、交付物、代码位置、复现命令都能在文件里找到。
- [ ] **路径真实**：`实现计划`/`交付物` 里的代码和产物路径确实存在，或明确标 TBD。
- [ ] **命令能跑**：产出/验证命令可直接粘贴运行，不是伪代码。
- [ ] **不抄 runs/**：没有把 git_commit/env/data/精确命令抄进 task 文件，而是指向 `run_config.json`。
- [ ] **状态准确**：`状态` 和 `最近更新` 反映真实进展；`已完成` 的任务 `交付物` 节已填。
- [ ] **可验收**：`验收标准` 是可勾选、可客观判断的条件，不是"做好为止"这类空话。
- [ ] **逐条绑验证**（治谎报完成）：`验收标准` 每条都内联了一条**可执行**的「验证」命令，且注释了期望输出（true / exit 0 / 指定 artifact）——不是只在文末堆一坨 grep。
- [ ] **有切片计划**（治漂移）：`Checkpoint 切片` 列了 2–5 个有序切片，每片绑一条验收标准 + 一个报到点。
- [ ] **有范围栅栏**（治理解偏差/过度工程）：`阻塞与边界` 写了 `本任务不做（Non-goals）` 与越界处置，且有 `## 开放问题` 收口。
